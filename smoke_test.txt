We want a small smoke test for motion-aware subset selection on PointOdyssey, before building any game-data plumbing.

Goal

Test whether curating candidate training pairs/windows using motion-aware selection beats:
	•	random selection
    	•	current heuristic selection

        under a fixed retained-data budget.

        This is a canary, not the full paper. Keep it small, offline, and easy to iterate.

        Core hypothesis

        Dense candidate pairs are redundant.
        Bad stride / motion-threshold settings can produce poor training subsets.
        A simple balanced motion-aware selector should outperform:
        	•	random retention
            	•	current stride+motion heuristic
                	•	maybe simple motion novelty

                    especially under smaller budgets.

                    Source data

                    Use an oversampled PointOdyssey candidate pool.

                    We want to build a large candidate set from PointOdyssey using permissive settings, for example:
                    	•	multiple strides
                        	•	relaxed motion thresholds
                            	•	maybe multiple candidate windows per trajectory

                                The point is to create a redundant source pool that we later prune offline.

                                Pooled eval bank

                                We also want a pooled eval motion bank built from several downstream eval datasets.

                                For the smoke test, keep this simple:
                                	•	sample motion descriptors from several eval datasets
                                    	•	merge them into one pooled target bank
                                        	•	normalize consistently with the source descriptors

                                            This pooled bank is only for the target-aware selector variants.

                                            Descriptor space

                                            Use the motion descriptors from the previous paper.

                                            Likely start with one descriptor only for simplicity:
                                            	•	first choice: whatever is easiest / already implemented
                                                	•	BFV or HOF is fine

                                                    Do not start with multiple descriptor families unless almost free.

                                                    Selection policies

                                                    Implement these offline subset policies:

                                                    1. Random

                                                    Uniform random subset of candidate pairs.

                                                    2. Current heuristic

                                                    Use the current hand-tuned pair extraction rule:
                                                    	•	stride choice
                                                        	•	motion threshold choice
                                                            	•	whatever current baseline you actually use

                                                                This is important because the whole motivation came from brittle heuristic settings.

                                                                3. Motion novelty

                                                                Select pairs based on symmetric diversity in motion space.
                                                                Simple version:
                                                                	•	farthest-first in descriptor space, or
                                                                    	•	accept if distance to selected bank exceeds threshold

                                                                        No pooled eval support term here.

                                                                        4. Balanced motion-aware selector

                                                                        This is the main proposed method.

                                                                        Score candidates using:
                                                                        	•	pooled target support
                                                                            	•	motion novelty / breadth
                                                                                	•	redundancy control

                                                                                    Simple conceptual form:

                                                                                    score(x) = lambda_support * support(x)
                                                                                    + lambda_novelty * novelty(x)
                                                                                    - lambda_redundancy * redundancy(x)

                                                                                    For the smoke test, keep this implementation simple and greedy.

                                                                                    Recommended support definition

                                                                                    For the balanced selector, define support roughly as:
                                                                                    	•	candidate covers pooled eval points that are currently undercovered by selected training subset

                                                                                        Approximation is fine. It does not need to be mathematically fancy yet.

                                                                                        For example:
                                                                                        	•	find nearest pooled eval descriptors to candidate
                                                                                            	•	reward candidate if it is close to pooled eval regions not yet well covered by current selected bank

                                                                                                Recommended novelty definition

                                                                                                Use simple motion-space novelty:
                                                                                                	•	distance to nearest selected training descriptor
                                                                                                    	•	or farthest-first style greedy coverage

                                                                                                        Recommended redundancy definition

                                                                                                        Can be simple:
                                                                                                        	•	same as negative nearest-neighbor distance to selected bank
                                                                                                            	•	or just rely on novelty thresholding if you want to avoid an extra term

                                                                                                                Budgets

                                                                                                                Use exactly three subset budgets:
                                                                                                                	•	5%
                                                                                                                    	•	15%
                                                                                                                        	•	30%

                                                                                                                            of the oversampled candidate pool

                                                                                                                            If absolute counts are easier, that is also fine, but keep three levels:
                                                                                                                            	•	low
                                                                                                                                	•	medium
                                                                                                                                    	•	high

                                                                                                                                        Run grid

                                                                                                                                        Main smoke-test grid

                                                                                                                                        Run these methods at each budget:

                                                                                                                                            	•	Current heuristic
                                                                                                                                                	•	Motion novelty
                                                                                                                                                    	•	Balanced
